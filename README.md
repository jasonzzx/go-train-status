# GO Train Status

A mobile-first PWA for GO Transit rail commuters. Supports all 7 GO rail lines (Lakeshore West, Lakeshore East, Stouffville, Richmond Hill, Barrie, Kitchener, Milton) — pick your line and home station, the office terminus is always **Union Station**. Shows live departure schedules, real-time platform/delay status, service alerts, an interactive Union Station platform map, and expandable stop-by-stop breakdowns. Available in English and Simplified Chinese.

Live: https://go-train-status.vercel.app/

---

## Tech Stack

- **Next.js 14** (App Router, `'use client'` page)
- **TypeScript**
- **Tailwind CSS** with GO Transit colour palette
- **Vercel** for hosting (auto-deploys from GitHub `main`) + optional Vercel KV (Upstash Redis) for server-side platform memory
- **cheerio** for the legacy HTML-scraping fallback paths

## Data Sources

### Line Registry (`src/lib/lines.ts`)
Per-line display metadata (English + Chinese name, colour, default home station) plus the GTFS-derived station list and schedule for each of the 7 GO rail lines, loaded from `src/data/schedules/*.json` (generated by `scripts/generate-schedule.mjs` from the official GO GTFS feed). Every line's station list ends at Union Station — the app treats `UN` (Union) as the fixed office terminus regardless of which line/home station the user picks.

### Schedule (`src/lib/schedule-data.ts`)
Line-agnostic trip + stop-sequence logic built on top of `lines.ts`. Given a `lineId`, returns trips for any station/direction/service type (weekday/Saturday/Sunday) on that line. `vehicleType: 'train' | 'bus'` distinguishes train trips from express bus trips (direct, no intermediate stops).

### Live Tracker (`/api/tracker`) and Service Alerts (`/api/alerts`)
Both routes try the **official Metrolinx Open Data API** first when `METROLINX_API_ENABLED=1` and `METROLINX_API_KEY` are set (see `src/lib/metrolinx/`), and fall back to legacy HTML scraping on any error so behaviour never regresses:

- **Tracker**: `src/lib/metrolinx/trains.ts` merges `ServiceataGlance/Trains/All` (delay, cars), `ServiceUpdate/UnionDepartures/All` (outbound platform), and `ServiceUpdate/Exceptions/Train` (cancellations) into a `tripNumber → LiveTrainStatus` map, joined onto the GTFS schedule by trip number (`tripId.split('-').pop()`). Inbound (home-station) platform comes from a per-trip `Schedule/Trip/{date}/{tripNumber}` call. Fallback: scrapes [railsix.com](https://railsix.com)'s embedded SvelteKit trip data for the selected line's home station.
- **Alerts**: `src/lib/metrolinx/alerts.ts` calls `ServiceUpdate/ServiceAlert/All`, filtered by line code (amenity/elevator notices are dropped). Fallback: scrapes gotransit.com's service-updates page, then railsix.com/alerts.
- All official calls go through `src/lib/metrolinx/client.ts`: a 300/sec token-bucket rate limiter, a short-TTL in-memory cache with in-flight de-dupe (so the 30s client poll collapses to one upstream call), and `metrolinxLineCode()` to map app line ids to API line codes (Kitchener is `GT` in the live feed).

See `docs/api-migration-plan.md` for the full mapping notes, known gaps, and verification history.

### Platform memory (`src/lib/platform-store.ts`, `src/lib/platform-cache.ts`)
GO only publishes a Union departure's platform ~10–15 minutes before departure and wipes it from every API the instant the train leaves, so an already-departed ("NOW") trip has no platform anywhere. Two complementary caches paper over this:
- **Server-side** (`platform-store.ts`): writes to Vercel KV / Upstash Redis (`KV_REST_API_URL`/`KV_REST_API_TOKEN`, or `UPSTASH_*`), keyed per service day with a 30h TTL. Populated by `GET /api/cron/platforms` (protected by `CRON_SECRET`), intended to be hit every ~5 min by an external scheduler (e.g. cron-job.org). `/api/tracker` reads it to backfill departed trips for *any* device.
- **Client-side** (`platform-cache.ts`): a localStorage fallback scoped to the current browser and service day, so a platform sticks around through departure even without the server-side store configured.

Both no-op safely if their respective env vars aren't set.

## Internationalization (`src/i18n/`)
`LanguageProvider` (wrapped around the app in `layout.tsx`) holds the active language (`en` | `zh`, persisted to localStorage) and exposes `useLanguage() → { lang, setLang, t }`. UI strings live in `en.json` / `zh.json`; `t(key, vars?)` looks up the active dict and falls back to English, with `{var}` interpolation. Station names are translated separately via `station-names.zh.json`. Visiting `/en` or `/zh` sets the stored preference and redirects to `/` — useful as a bookmarkable/shareable link.

## Union Station Platform Map (`src/components/UnionPlatformMap.tsx`)
An SVG schematic of Union's Lower Level / concourses (York/VIA Concourse Halls, York St. and Bay St. covered walkways), redrawn from the City of Toronto Union Station Map PDF. Each of platforms 3–13 is drawn as its own labelled, highlightable segment (`src/lib/union-platform-map.ts` parses multi-platform tracker strings like `"12, 13"` and maps them onto the diagram). When a trip's platform falls in that range, the schedule card shows the map with the relevant platform(s) pulsing and an optional zoomed-in view.

## Intermediate Stops

Each line's full outward-to-Union station list (including intermediate stops) comes from `lines.ts` / `src/data/schedules/*.json`. Bus trips are direct (no intermediate stops); train trips stop at every station on the line between the selected home station and Union.

When `trackerTrips` includes a `stops: string[]` for a trip (live data from railsix), those station names override the GTFS-derived list via the `liveStops` prop on `TrainCard`. The official Metrolinx path doesn't supply a live stop list, so trips sourced from it fall back to the static GTFS stops.

## Auto-Update on Deploy

Since this is an installed PWA (standalone mode on iOS/Android), browsers can keep serving a stale cached build indefinitely after a new deploy. To fix that:

- `scripts/write-version.mjs` runs as a `prebuild` step and writes `public/version.json` with a `{timestamp}-{gitSha}` version string on every build.
- `next.config.mjs` sets `Cache-Control: no-cache, no-store, must-revalidate` on `/`, `/manifest.json`, and `/version.json` so the HTML shell and version probe are never cached.
- `layout.tsx` reads `public/version.json` at build time and passes it into `VersionWatcher`, a client component that polls `/version.json` every 60s (plus on tab focus/visibility change) and calls `window.location.reload()` if the server's version no longer matches the one baked into the page.

Result: any open tab or installed PWA picks up a new deployment automatically, without the user needing to force-quit and relaunch the app.

## Key Features

- **Line + home station picker** — switch between any of the 7 GO rail lines and their stations; office terminus is always Union Station
- **English / Simplified Chinese toggle** — translates UI strings and station names, persisted to localStorage
- **Direction toggle** — Home → Union / Union → Home
- **Date picker** — browse any date's schedule (weekday/Saturday/Sunday service)
- **"Next" badge** — highlights the next departure
- **"Now" badge + On Board** — amber pulsing badge and expandable progress view for in-progress trains
- **Expandable stop list** — shows intermediate stations with estimated times and a live position indicator
- **Platform + delay badges** — from the official Metrolinx API (railsix fallback), refreshed every 30 seconds, with server- and client-side caching so a platform survives a train's departure
- **Union Station platform map** — tap a platform badge to see it highlighted on an interactive concourse map
- **Service alerts sheet** — bottom sheet with GO Transit alerts, line-filtered
- **Line status badge** — Good Service / Minor Delays / No Scheduled Service, derived from live tracker data
- **Install guide sheet** — detects iOS Safari vs Android Chrome and walks the user through "Add to Home Screen"
- **PWA** — installable, standalone display mode, custom icon

### Environment variables (all optional — the app runs with sensible fallbacks when unset)

| Variable | Purpose |
|---|---|
| `METROLINX_API_KEY` | API key for the official Metrolinx Open Data API |
| `METROLINX_API_ENABLED` | `1`/`true` to use the official API (with key set); otherwise routes use the legacy scrapers |
| `CRON_SECRET` | Shared secret required by `GET /api/cron/platforms` (`Authorization: Bearer …` or `?secret=`) |
| `KV_REST_API_URL` / `KV_REST_API_TOKEN` | Vercel KV (or `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN`) for the server-side platform store |

Deploy is via `git push` to `main` — Vercel auto-deploys from there. To finish enabling the official API path in production, set `METROLINX_API_KEY`/`METROLINX_API_ENABLED` (and the KV vars, if using the platform store) in the Vercel project's environment, then redeploy; see `docs/api-migration-plan.md` for the rollout checklist.
